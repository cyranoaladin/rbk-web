Build an AMM

You will build a minimal constant-product AMM (x · y = k) with audit-grade protections: strict vault ownership, deterministic PDAs, explicit fees, and min_out slippage guards.

Outcome
  • Understand pool state, vaults, LP mint, and invariant enforcement.
  • Prevent classic AMM failures: vault spoof, decimals mismatch, fee leakage, slippage abuse.
  • Ship proofs: negative tests for swaps, deposits, withdrawals, and CPI boundaries.

Invariant-first mindset

  • State invariants: vault mints/owners are fixed; LP mint is fixed; fees are consistent.
  • Math invariants: constant-product is maintained up to rounding; no negative balances.
  • User invariants: user can bound outcomes via min_out and explicit errors.

Minimal constant-product model

Let x be vault A reserve and y be vault B reserve. A swap enforces x · y = k up to rounding, after applying fees.

Non-negotiables
  • Vaults are owned by the program (PDA), not by users.
  • Vault mints never change after initialization.
  • LP mint supply and user LP balances are consistent with deposits/withdrawals.
  • Swap output must respect min_out (slippage guard).
Typical failures
  • Decimals mismatch leading to wrong pricing.
  • Vault spoof (attacker passes their own token account as “vault”).
  • Fee leakage due to inconsistent rounding or missing checks.
  • Missing min_out leading to user value loss on volatile price movement.

Module validation

  • Pool init, deposit, withdraw, swap implemented with explicit invariants.
  • Negative tests exist for vault spoof, mint mismatch, min_out violations, and fee edge cases.
  • QA is green; navigation and bilingual routing are correct.
