Build an AMM

Vous construisez un AMM constant-product minimal (pool + LP + swaps) pour apprendre les invariants, la gestion des arrondis et les patterns CPI Token. Le but est pédagogique : écrire quelque chose d’auditable, testable, et robuste.

Objectif
  • Comprendre x·y = k, slippage, fees, LP shares.
  • Faire des maths sûres : u128, arrondis, checked_*.
  • Hardening : vault checks, program id checks, invariants testés.

Invariant central

Dans un AMM constant-product, la relation x·y ≈ k (avec fees et arrondis) doit rester cohérente. La validation “audit-grade” consiste à formaliser les invariants et à les rendre falsifiables par tests.

Checklist AMM (audit-grade)

Vaults & accounts
  • Vault A/B : owner + mint vérifiés (pas de substitution possible).
  • PDA authority : seeds + bump contraints, signer CPI contrôlé.
  • LP mint : supply cohérente, authority contrôlée, mint attendu.
Math & invariants
  • Calculs en u128, puis cast explicite (et tests limites).
  • Arrondis définis : “round down” pour protéger le pool, “min_out” pour protéger l’utilisateur.
  • Invariants testés : conservation, bornes, monotonicité, pas de “création” de valeur.

Validation du Module

Pour valider l’AMM, vous fournissez : add/remove, swaps, tests d’arrondis, et un bloc “invariants”.
