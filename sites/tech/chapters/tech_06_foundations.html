01

FoundationsThéorie Solana (Avant Anchor)

Cette page installe le modèle mental Solana. L’objectif est de comprendre ce qui se passe réellement dans une instruction : quels comptes sont passés, avec quels droits (signer/writable), quelle donnée est lue/écrite, et comment un programme peut dériver une adresse (PDA) et appeler un autre programme (CPI).

Pré-requis

  • Notions de base : hash, signature, sérialisation.
  • Rust minimal (lire types, Result/Option). Pas besoin d'Anchor.

Objectifs Opérationnels

1. Expliquer l'Account Model (owner, data, lamports).
2. Lire une Tx : instruction, accounts metas, signer/writable.
3. Maîtriser les PDAs (seeds, bump, authority).
4. Comprendre les CPIs (invoke vs invoke_signed, security).
5. Modèle Mental SPL Token (Mint, Account, Authority).
01

Accounts : Structure & Invariants

  • Owner Le Programme ID qui contrôle le compte. Seul lui peut modifier les Data ou débiter les Lamports.
  • Data Tableau d'octets brut (`Vec`). Le programme définit la structure (Borsh, etc).
  • Lamports Solde du compte. Paye le loyer (Rent). Si 0, le compte est purgé.
  • Mutability Règle d'or : Un compte passé en lecture seule ne PEUT PAS être modifié.

PDA : Program Derived Address

Adresse Déterministe

Pas de clé privée. Dérivée via `hash(seeds, program_id)`.

Address as Authority

Le programme peut "signer" pour cette adresse via `invoke_signed`.

04

CPI : Cross-Program Invocation

invoke

Appel standard. Le programme délègue ses droits de signature reçus.

Program A --(sign A_user)--> Program B

invoke_signed

Magie PDA. Le runtime vérifie les seeds et injecte la signature du PDA.

Program A --(sign PDA)--> Program B
⚠️ RISQUE : Toujours whitelister le Program ID cible + valider les comptes passés.
