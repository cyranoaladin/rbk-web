Performance : Ce que le Builder doit savoir

Compute Units (CU)

Chaque instruction a un "coût". Trop de logs, trop de CPIs, ou de la mauvaise crypto = Transaction Failed (ComputeBudgetExceeded).

Priority Fees

Quand le réseau est congestionné, payer plus (Micro-lamports) permet de passer devant. C'est une enchère de blockspace.

Logs Pollution

`msg!` coûte cher (syscall). En prod, on log l'essentiel, pas le debug.

Attention aux boucles infinies simulées Itérer sur un grand nombre de comptes dans une seule transaction est un anti-pattern sur Solana. La limite de CU vous rattrapera. Préférez une architecture "off-chain indexer + stateless logic" ou du chunking.

Ops & Livrables

Bonnes Pratiques Ops

  • Déploiements : Toujours vérifier le buffer authority avant upgrade.
  • Rollback : Avoir un plan (e.g., remettre le binaire précédent) si la migration state échoue.
  • Post-Mortem : Si ça casse, on écrit pourquoi. Sans blâme, mais avec précision.

Exercice : Panne Sèche

"Vos utilisateurs se plaignent que 'le site ne marche pas'. Votre RPC répond timeout. Le validateur semble up. Que faites-vous ?"

1. Switcher de RPC (Failover).
2. Vérifier Solana Beach / Solscan (Cluster status).
3. Si Cluster downetc. on attend (et on communique).
