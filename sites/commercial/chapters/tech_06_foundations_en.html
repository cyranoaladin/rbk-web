FoundationsSolana Theory (Pre-Anchor)

This page establishes the **Solana mental model**. The goal is to understand what really happens inside an instruction: which accounts are passed, with what privileges (signer/writable), what data is read/written, and how a program can derive an address (PDA) and call another program (CPI).

Prerequisites

  • Basic concepts: hash, signature, serialization.
  • Minimal Rust (reading types, Result/Option). No Anchor needed yet.

Operational Objectives

1. Explain the Account Model (owner, data, lamports).
2. Read a Tx: instruction, accounts metas, signer/writable.
3. Master PDAs (seeds, bump, authority).
4. Understand CPIs (invoke vs invoke_signed, security).
5. SPL Token Mental Model (Mint, Account, Authority).
01

Accounts: Structure & Invariants

  • Owner The Program ID that controls the account. Only it can modify Data or deduct Lamports.
  • Data Raw byte array (`Vec`). The program defines the structure (Borsh, etc).
  • Lamports Account balance. Pays for storage (Rent). If 0, the account is purged.
  • Mutability Golden Rule: An account passed as Read-Only CANNOT be modified.

04

CPI: Cross-Program Invocation

invoke

Standard call. The program delegates its signature privileges.

Program A --(sign A_user)--> Program B

invoke_signed

PDA Magic. Runtime checks seeds and injects PDA signature.

Program A --(sign PDA)--> Program B
⚠️ RISK: Always whitelist the target Program ID + validate passed accounts.
